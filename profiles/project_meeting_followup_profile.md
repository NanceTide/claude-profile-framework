# 项目会议后续跟进 Profile

## 适用场景

我说「整理会议并发后续邮件」「会议后帮我同步一下」「把这次会整理成纪要再写封邮件」「会后跟进」时启用。

如果我只说「整理会议纪要」，只使用会议纪要 Skill，不进入邮件生成。

如果我只说「帮我回邮件」或「写封邮件」，只使用邮件回复 Skill，不强行补会议纪要。

若同时像「写周报」又像「会后跟进」，先列出候选 Profile 让我选择。

## 技能依赖

**必选：** 会议纪要 Skill、邮件回复 Skill。

本场景必须串行使用多个 Skill，不并行执行。会议纪要是邮件草稿的上游输入。

## 目标标准

目标是把一次项目会议变成两份可直接使用的交付物：

- 一份结构化会议纪要，能给未参会者快速理解背景、结论和行动项
- 一封会后同步邮件，能明确告诉相关人下一步做什么、谁负责、什么时候完成

必须覆盖：会议概要、讨论要点、行动项、会议决议、邮件正文。

行动项必须包含负责人和截止时间；缺失时标「待确认」或向我追问，不编造。

**取舍原则：** 忠实纪要优先于邮件润色。若会议素材与邮件诉求冲突，以会议素材为事实来源，并提醒我确认。

## 流程编排

1. **串行：会议纪要 Skill。**
   - 输入：会议录音转写、会议笔记、聊天记录、口述要点或用户提供的会议材料。
   - 输出：会议概要、讨论要点、行动项、会议决议。
   - 若会议主题或素材缺失，先追问，不进入邮件生成。

2. **检查点：纪要确认。**
   - 展示会议纪要摘要和行动项表。
   - 如果行动项中有「待确认」负责人或截止时间，先让我确认是否继续。

3. **串行：邮件回复 Skill。**
   - 输入：已整理的会议纪要、行动项表、用户指定的收件对象与邮件目的。
   - 输出：会后同步邮件正文。
   - 若用户未指定收件对象，按「参会人 + 需要执行行动项的人」生成泛用版本，不写 To/CC。

4. **汇合：最终交付。**
   - 先输出完整会议纪要。
   - 再输出可直接复制的会后同步邮件正文。
   - 最后列出仍需用户确认的字段。

## 协作规范

**检查点：**

- 会议纪要 Skill 完成后必须暂停一次，先让我确认行动项。
- 我确认后再启动邮件回复 Skill。
- 若我明确说「不用确认，直接写邮件」，可跳过暂停，但邮件中必须保留「待确认」字段。

**汇报：**

- 中间只发纪要摘要和行动项表，不发完整分析过程。
- 最终一次性交付纪要和邮件。

**异常：**

- 会议素材太乱：先按议题归并，让我确认分类后再写纪要。
- 行动项缺负责人或截止时间：尽量从素材推断；推断不出标「待确认」。
- 邮件收件人身份不清：默认写成内部协作邮件，语气简洁专业。
- 涉及敏感信息：照常整理，但在最终提示我发送前脱敏。

## 知识预设

自动从 Memory 取用：

- 我的姓名、部门、常用署名
- 当前项目名（如 Memory 或当前对话中已有）
- 常用邮件语气偏好（如有）

不从 Memory 取用：

- 本次会议事实、结论和行动项；这些必须来自用户本次提供的素材
- 本次邮件具体收件人；除非对话或纪要中明确出现

值得写入 Profile Memory 的事实：

- 常用会后邮件署名
- 常用收件群组或同步对象
- 固定会议纪要模板偏好

## 交付规范

- 输出格式：Markdown
- 先给「会议纪要」，再给「会后同步邮件」
- 会议纪要包含：会议概要、讨论要点、行动项、会议决议
- 邮件正文不带「以下是」前缀，不输出邮件头，除非我要求
- 邮件不超过 250 字，语气简洁、明确、行动导向
- 不编造会议中没有出现的事实
