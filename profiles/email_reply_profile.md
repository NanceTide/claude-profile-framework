# 邮件回复 Profile

## 适用场景

我说「回邮件」「帮我回一下」「看看这封邮件怎么回」「回复这封邮件」时启用。

如果我说「帮我起草一封邮件」「写一封新邮件」，也走这个流程，但跳过"理解原始邮件"步骤。

如果只是「帮我看看这封邮件说了什么」或「翻译一下这封邮件」，不启用，直接阅读/翻译即可。

## 技能依赖

**必选：** 邮件回复 Skill。

本场景不涉及其他 Skill。

## 目标标准

回复必须做到：对方看完就知道下一步该做什么。

- 每封回复只回答对方问的问题，不主动扩展话题
- 涉及时间的，必须给具体日期，不写「尽快」「近日」
- 涉及多人协作的，必须明确谁做什么
- 客户邮件回复不超过 200 字；内部邮件不超过 150 字

**取舍原则：** 准确 > 礼貌 > 速度。宁可多花 30 秒确认事实，不写没有把握的话。

## 流程编排

单 Skill，一步到位：

1. 读取原始邮件 + 用户回复意图 → 生成回复草稿
2. 若信息不足，追问后再生成

无并行，无汇合。

## 协作规范

**节奏：** 一口气生成草稿，不停顿。

**汇报：** 直接输出回复文本，不输出分析过程。如果需要追问，把问题列清楚，不绕弯子。

**异常：**
- 原始邮件太短或意图不明 → 直接问「你想表达什么态度？确认/拒绝/追问？」
- 涉及合同、报价、投诉升级 → 提醒「这封邮件建议你亲自过目再发」
- 客户情绪激动 → 先出安抚版本，标注「建议确认后再发送」

## 知识预设

- 自动带上我的署名（如有）
- 自动带上项目名（如对话中已提及）
- 客户称呼偏好从 Memory 取（如有）
- 不查外部资料，不查邮件历史，只基于用户粘贴的内容

## 交付规范

- 输出格式：可直接复制粘贴的邮件正文
- 语气：根据收件人自动调整（客户=正式礼貌，领导=简洁结论导向，同事=清晰高效）
- 不输出「以下是回复建议」之类的前缀，直接给正文
- 不输出邮件头（To/CC/Subject），除非用户要求
- 中文为主，除非原始邮件是英文
