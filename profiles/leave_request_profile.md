# 请假申请 Profile

## 适用场景

我说「我要请假」「帮我请假」「提交请假申请」「请年假/事假/病假/调休」「帮我给领导发请假」时启用。

如果我只是问「年假怎么算」「病假要什么材料」「还剩多少假」，先按咨询回答，不直接生成申请。

如果我同时要求「请假并写邮件」，仍以本 Profile 为主；先完成请假申请字段，再输出可发送给领导或 HR 的文本。

## 技能依赖

**必选：** 提交请假申请 Skill。

本场景不依赖其他 Skill。若需要邮件正文，由请假申请 Skill 按本 Profile 的交付规范生成，不额外调用邮件回复 Skill。

## 目标标准

目标是生成一份可以直接提交到公司系统或发给审批人的请假申请。

必须确认：

- 请假人姓名
- 部门
- 工号
- 请假类型
- 开始时间
- 结束时间
- 请假时长
- 请假事由
- 审批人

请假理由要真实、简洁、不过度披露隐私。提交前不得保留关键字段空缺。

**取舍原则：** 准确优先于速度；缺关键字段时宁可暂停追问，也不生成看似完整但事实不明的申请。

## 流程编排

单 Skill，串行执行：

1. 识别用户是否要生成或提交请假申请。
2. 从当前对话抽取请假类型、时间、时长、事由、输出形式。
3. 从 Profile Memory 自动带上工号、部门、直属领导、常用审批人等稳定信息。
4. 检查关键字段完整性；缺少请假类型、时间、时长、事由、审批人时必须追问。
5. 生成请假表单字段和一段可发送文本。
6. 如果用户要求提交、发送或转交，先展示最终摘要并等待「确认提交」。
7. 用户提供或更正工号、部门、领导、审批人、系统入口等长期复用信息时，写入本 Profile 的「场景记忆」区。

## 协作规范

**节奏：**

- 信息足够时一口气生成申请内容。
- 信息不够时每次只追问最关键的 1-3 个字段。
- 涉及提交、发送、转交时必须暂停确认。

**汇报：**

- 草稿阶段输出表单字段 + 发送文本。
- 提交前输出「请假类型、时间、时长、审批人、抄送人」摘要。
- 不输出内部推理过程。

**异常：**

- 「明天」「下周一」等相对日期，要换算为具体日期，并在输出中写明。
- 剩余年假、调休余额无法核验时标「待核验」，不估算。
- 病假、婚假、产假、丧假等可能需要证明材料时提醒我补附件。
- 审批人和 Profile Memory 冲突时，以我当前对话最新说明为准，并更新场景记忆。

## 知识预设

执行本 Profile 时，自动从 Profile Memory 取用并填入：

- 工号
- 部门
- 岗位
- 直属领导
- 常用审批人
- 常用抄送人
- 常用工作交接人
- 常用请假系统入口

这些信息都属于请假场景高复用事实：如果我在对话中提供或更正它们，请写入本 Profile 的「场景记忆」区，未来无需重复询问。

不要写入 Profile Memory：

- 本次请假日期
- 本次请假时长
- 本次请假理由
- 本次临时备注
- 一次性抄送对象

如果 Profile Memory 中的剩余年假或调休余额超过一次执行未刷新，视为可能过期，提交前提醒我核验。

## 交付规范

默认输出两部分：

```yaml
请假人:
部门:
工号:
请假类型:
开始时间:
结束时间:
请假时长:
请假事由:
审批人:
抄送人:
工作交接:
备注:
```

以及一段可直接发送的中文请假文本。

语气简洁、正式、自然，不写过度客套话。请假理由控制在一句话内。若是发给直属领导，用「请您审批」；若是系统表单，用字段化表达。
