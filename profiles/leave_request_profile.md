# 请假场景 Profile

## 适用场景

当我说以下或类似内容时启用：

- 「帮我请假」「请假」「提交请假」
- 「明天请假」「下周一请个假」「想请几天假」
- 「身体不舒服请个病假」「请年假」
- 在对话中直接说了请假意图（如「xx 号到 xx 号请假，批一下」）

若我只是问「我还有多少天年假」「请假流程是什么」「病假需要什么材料」，不启用，先回答/解释即可。

单个 Profile，无多场景冲突问题。

## 技能依赖

**必选：** 请假申请 Skill（leave_request_skill）。

本场景不涉及其他 Skill。

## 目标标准

请假申请必须覆盖以下字段：

- **必填：** 姓名、部门、请假类型、开始时间、结束时间、天数、审批人、工作交接
- **选填：** 事由（简洁即可，不强制详细）

**核心追求：**

- 申请文本**控制在一小段内**，不要分点、不要表格、不要分段展开
- 长度控制在 3–6 行（微信/钉钉/飞书的日常请假风格）
- 关键字段一个不漏，但要自然融入段落中
- 语气简洁直接，不过度客气（适合跟直属领导日常沟通的风格）

**完成定义：** 生成一段可复制发送的请假文本，用户确认或指示修改。

## 流程编排

单 Skill，一步到位：

1. 从对话中抽取请假信息，同时从 Memory 补全身份信息（姓名、部门等）
2. 若所有字段齐全 → 直接生成请假文本草稿，展示给用户
3. 若缺少必填字段 → 追问缺失信息，补齐后生成草稿
4. 用户确认或要求修改后，输出最终文本

无并行，无汇合。

**分支：** 若用户明确说「你先写，不完整也行」或类似意思，按「协作规范」的放行规则处理。

## 协作规范

**节奏：**
- 用户表达了请假意图 + 部分信息 → 先补齐信息，再生成草稿。不中断追问每一个字段，能一次问清就一次问清
- 用户说「你先写差不多的就行」或「你看着办」或「直接出吧」→ 用「待补充」占位缺失信息直接生成，把草稿先亮出来

**汇报：**
- 第一次产出时直接展示请假文本，不加分析，不加"以下是我整理的"等前缀
- 如果追问，用简洁的问题列表，一个问题一行

**放行规则（关键）：**
- 正常流程：草稿生成后，必须确认至少以下字段齐全：姓名、请假类型、起止时间、天数、审批人
- **简略模式**：若用户说「直接出吧」「差不多就行」「先写」「你先出」或明确表示不用追全，用「待补充」占位缺失的字段，直接输出文本。不在输出前追问。
- 用户确认后直接输出最终文本，不再额外确认。

**异常：**
- 日期含糊（「明天」「下周一」「月底」）→ 换算为具体日期并标注，让用户确认
- 审批人/部门信息 Memory 里取不到 → 追问
- 涉及病假 → 提醒「涉病假建议按公司制度准备假条或诊断证明」

## 知识预设

**来源优先级：** 当前对话提取 > Profile Memory 读取 > 追问用户

**自动从 Memory 取用**（以下事实已满足持久化标准，存入 Profile Memory）：

- 姓名
- 部门
- 常用审批人（直属领导）

> 首次执行时若 Memory 中无这些信息，追问用户，确认后写入 Profile Memory。

**不查外部资料：** 不查考勤系统、不查请假余额、不查公司制度。以上信息如用户需要，可以说明但不会自动查询。

**查不到时的占位规则：**
- 信息不全但用户同意「直接出」→ 用「待补充」占位
- 信息不全且用户未放行 → 追问补齐再出

## 交付规范

- **输出格式：** 一段简洁的文本，可直接复制粘贴发送。不加前后缀、不加 Markdown 格式、不加"以下为……"等引导语
- **篇幅：** 3–6 行，整段书写（不分点、不列表）
- **风格示例（这就是要的效果）：**

  > 王强-产品部，申请 6 月 10 日（周三）至 6 月 12 日（周五）事假 3 天，工作已交接给赵四。审批人：李总。请批准。

- **语气：** 简洁直接，日常沟通风格，不过度客气
- **禁忌：** 不分条列点、不写长句、不用「尊敬的」「您好」「此致」等正式敬语、不加表情符号
- **确认用语：** 展示文本后说「这样可以吗？」或「要修改吗？」即可，不说「请确认是否提交」之类正式措辞
