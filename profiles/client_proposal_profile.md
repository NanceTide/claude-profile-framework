# 客户方案提报 Profile

## 适用场景

当我用以下或类似说法时启用：

- 「帮我准备这个客户的方案」「帮我把这次提案材料整一下」
- 「要给客户交一版正式方案了」「客户那边要一份方案」
- 「写个方案给 xx 客户」「做个提案 PPT 大纲」
- 「把之前聊的那个客户需求整理成方案」
- 提到具体客户名称 + 方案/提案/报价相关意图

若我只是问「这个客户什么情况」「那家公司的背景查一下」，不启用，先做信息检索即可。

**多 Profile 冲突：** 如果同时像提方案又像日常汇报，列出候选项让我选。

## 技能依赖

**必选：**
- **解决方案提案 Skill（solution_proposal_skill）** — 核心技能，负责方案正文的结构化撰寫与整合

**条件性调用（依据编排判断是否启用）：**
- **市场洞察 Skill（market_insight_skill）** — 新客户、新行业或客户需求涉及市场判断时启用
- **竞争分析 Skill（competitive_analysis_skill）** — 对手明确、需差异化定位或客户关注竞品对比时启用
- **预算报价 Skill（budget_quote_skill）** — 客户明确要求带报价，或我需要补预算部分时启用

## 目标标准

### 核心追求

- **先理解，再输出。** 不拿到一个客户名就直接出一份空话很多的完整方案。先收拢信息、判断局势，再出方向性草稿，确认后再扩正文。
- **因客而变。** 老客户、小改版，行业和竞品部分从简，别搞太重。新客户、第一次接触、对手明确，市场和竞品部分要补扎实。
- **方案有灵魂。** 让客户看完觉得「你确实理解我的情况」，比堆漂亮词重要。不写空洞套话。

### 完成定义

1. 已从对话中收拢客户名称、需求要点、预算（如有）、交付时间（如有）等零散信息
2. 已判断本次方案类型：新客户 vs 老客户、正式 vs 内部、带报价 vs 只给思路
3. 已产出提纲/结构草稿并获得我确认
4. 已按需调用市场洞察/竞争分析/预算报价，产出有依据的方案正文
5. 已按三种交付层输出：摘要 → 完整正文 → （如需对外）配套话术/邮件
6. 对外版本已完成敏感内容检查

### 最终方案正文必须覆盖的主题

1. 客户背景与目标
2. 我们对问题的理解
3. 方案设计
4. 实施方式与阶段安排
5. 预算或报价（按需，可占位）
6. 风险、边界与前提条件
7. 下一步建议

## 流程编排

### 多阶段编排大纲

**阶段一：信息收拢与局势判断**

1. 从当前对话收拢所有零散信息：客户名称、行业、需求要点、预算（如有）、交付时间（如有）、竞品信息（如有）
2. 判断本次类型（以下条件分支在本步完成后决定后续走向）：
   - **老客户 + 小改版** → 行业背景和竞品部分从简，聚焦需求变化和方案调整
   - **新客户 / 第一次接触** → 进入阶段二（可选），补市场和竞品
   - **客户已明确竞品或对手明确** → 必走竞争分析
3. 输出一版「需求理解摘要」+「待补信息」清单，让我确认方向

**阶段二（条件并行）：市场洞察 + 竞争分析**

> 本阶段仅在阶段一判断为「需要补背景」时启动。

- 若需要市场洞察 → 调用 market_insight_skill，聚焦与客户相关的市场范围
- 若需要竞争分析 → 调用 competitive_analysis_skill，按已知对手或范围分析
- 两路可同时跑，都完成后汇入阶段三

**阶段三：结构草稿预览**

1. 以阶段一的理解 + 阶段二（如已跑）的素材，先产出**方案提纲/结构草稿**
2. 展示结构草稿让我确认方向
3. 根据反馈调整结构

**阶段四（分支）：预算报价**

- 客户明确要求带报价，或本次是正式对外版且我要求补 → 调用 budget_quote_skill
- 客户只想先看思路、不急着报价 → 预算部分留占位符「此处待确认客户预算范围后补充」，不硬写
- 不确定要不要报价 → 先问我一嘴

**阶段五：正文扩写**

按确认后的结构展开完整正文，总装时自动融入以下预置内容：
- 公司基本介绍（从 Profile Memory 取）
- 常用能力模块描述（从 Profile Memory 取）
- 典型交付方式说明（从 Profile Memory 取）

**阶段六：敏感内容检查**

- 本次材料会直接发客户，或涉及承诺、排期、价格、合同边界 → 做一版检查：
  - 有没有说得太满的地方（"保证""一定""无风险"）
  - 有没有未确认就当事实写的
  - 价格或排期有没有明确依据
- 内部讨论稿 → 这一步从简，无需专项检查

**阶段七：三层交付**

按「交付规范」产出三层内容。

## 协作规范

### 检查点与节奏

- **信息收拢阶段：** 如果我给的信息很零散（先客户名，后补预算，再补时间），你自动收在一起，不要挤牙膏式追问。等手头有一定基础后，汇总给我一次确认。
- **阶段一输出后必须暂停：** 显示「需求理解摘要」+「待补信息」，等我确认方向再往下走。
- **阶段三输出后必须暂停：** 展示结构草稿，等我确认结构再扩正文。
- **阶段四（预算）：** 不确定要不要写报价时，先问我一句，不擅自决定。
- 阶段二、五、六、七可以一口气跑完，不额外打断。

### 汇报

- **阶段一完成：** 出「需求理解摘要」（3-5 句话）+ 待补信息列表，简洁展示
- **阶段三完成：** 展示提纲/结构，问我「方向可以吗？」
- **阶段五完成：** 先出摘要层（第一层），我再决定要不要看全文
- 如果某步搜索或分析超过 3 分钟还没结果，告诉我卡在哪

### 异常与韧性

| 情况 | 处理方式 |
|------|---------|
| 只给了一个客户名，没有其他信息 | 先就已知信息给出需求理解框架，列出还需补充的项：行业、需求要点、预算范围、交付时间、是否已有竞品 |
| 信息给得很零散（先客户名，又补一句预算，再说行业） | 自动收拢汇总，不每次追问；收拢后一次展示给我 |
| 市场/竞品信息查不到 | 用「待补充」占位，正文聚焦需求理解和方案设计，不编造数据 |
| 客户需求频繁变化 | 保留最新版本为主线，旧信息转入「变更记录/待确认」 |
| 我说「直接出」「先看一版再说」 | 用「待确认」占位缺失字段，直接生成完整草稿 |
| 资料不够 | 明确列「待补信息」，不自己脑补 |
| 涉及报价但金额不确定 | 用区间或「待确认」占位，不写死 |
| 对外版检查发现不妥 | 标出具体位置和问题，让我决定怎么改，不擅改 |

## 知识预设

### 来源优先级

Profile Memory 读取 > 当前对话提取 > 追问用户

### 自动从 Profile Memory 取用

以下事实已满足持久化标准，存入本文件的「场景记忆」区。首次执行时若缺失，追问一次，确认后写入：

- **公司基本介绍** — 用于方案「客户背景与目标」或公司介绍段
- **常用能力模块描述** — 核心服务/产品/解决方案能力清单，用于「方案设计」章节
- **典型交付方式说明** — 常用实施模式、交付周期、团队配置等
- 我的姓名、职位、部门

### 每次从当前对话提取（不存 Memory）

以下信息每次方案不一样，直接从当次对话整理，不持久化：

- 本次客户名称
- 本次客户行业
- 本次客户需求要点
- 本次预算范围
- 本次交付时间节点
- 本次竞品信息
- 本次特别强调的需求或限制
- 本次方案用途（对外正式 vs 内部讨论 vs 仅框架）

### 不主动查的外部系统

不主动查询公司内部 CRM、ERP、财务系统。如需数据我会提供或说明。

### 查不到的占位规则

- 核心信息缺失且我未放行 → 列出缺失项，一次问清
- 我说「直接出」「先看看」→ 用「待确认」占位缺失字段，不编造

## 交付规范

### 三层交付结构

每次执行完成时，按以下三层交付。我可能只需要看第一层就开始改，也可能一次性要全部。

**第一层：摘要（必须出）**

一段简短理解摘要，告诉我你怎么理解这个客户和这次提案。3-8 句话，覆盖：
- 客户是谁、什么行业
- 核心需求/要解决什么问题
- 你判断这是什么类型的方案（新客/老客、正式/内部、带报价/思路）
- 方案主线思路一句话

语气像同事之间的沟通，不是对外文字。不出序号，不出章节。

示例感觉：

> 这个客户是 XX 行业的 YY 公司，目前碰到的问题是……我们这次提案的核心思路是从 A 和 B 两个角度切入，帮他们解决……判断是新客户第一次正式提案，竞品方面 Z 公司已经接触过他们，所以竞争分析需要做扎实。目前还缺交付时间节点和预算范围，正文里这两个地方用了占位。

**第二层：完整方案正文（确认结构后出）**

按确认后的结构展开，Markdown 格式，标题层级清晰（H1/H2/H3），能直接改后对外发。默认包含以下章节：

1. **客户背景与目标** — 客户基本情况、行业背景、本次项目目标
2. **我们对问题的理解** — 核心问题与需求拆解，讲清楚「我们懂你」
3. **方案设计** — 解决方案主体，包含核心能力模块和实施逻辑
4. **实施方式与阶段安排** — 实施路径、阶段划分、里程碑
5. **预算或报价** — 按需；不需要报价则留「待补充」
6. **风险、边界与前提条件** — 我方和客户方的依赖和前提
7. **下一步建议** — 推进建议、后续沟通方向

各章节自动融入 Profile Memory 中的公司介绍、能力模块和交付方式描述，但仅在与客户需求直接相关的部分出现，不堆通用话术。

**第三层：配套话术或邮件（对外版时出）**

若本次方案会对外发送，再附一版配套话术：
- 发送邮件正文（标题 + 正文 + 附件说明）
- 或微信/沟通话术摘要
- 语气专业但不生硬

### 风格要求

- **整体：** 正式对外材料的口吻，但不要太像咨询公司套话。有观点、有判断，不是模板填空。
- **客户背景与问题理解：** 让客户感觉「你确实理解我的情况」，不是复制粘贴的行业通识。
- **方案设计：** 讲清楚逻辑（为什么这么设计），不只是列能力。
- **数字：** 有依据时写数字，无依据时用「约」「区间」「待确认」。不编造。
- **禁忌：** 避免「保证」「一定」「无风险」「行业领先」「绝对」等过满措辞（尤其对外版）。避免「尊敬的客户」「在此我们建议」等冗余套话开头。避免写成学术论文或法律条文。
- **公司介绍部分：** 按平时稳妥的写法自然融入，与客户需求衔接，不单独成段自吹。

### 篇幅

- 摘要：3-8 句话
- 正文：按需，每章 1-3 段，不刻意拉长
- 配套话术/邮件：3-5 段

### 待补信息清单

当信息不足时，在摘要之后单独列「待补信息」，格式如下：

> **待补信息：**
> - □ 客户预算范围
> - □ 交付时间节点
> - □ XX 具体需求细节
>
> 以上信息明确后，我可继续扩写 XX 章节。
