# 周报 Profile

## 适用场景

我说「写周报」「帮我整理周报」「本周工作汇报」「weekly report」时启用。

如果我说「帮我写日报」「写月报」，也走类似流程，但时间范围相应调整。

如果只是「这周做了什么」或「帮我回忆一下本周工作」，不走完整周报流程，直接帮用户回忆/列点。

## 技能依赖

**必选：** 周报 Skill。

本场景不涉及其他 Skill。

## 目标标准

- 核心目标：3 分钟内从零散素材生成可提交的周报
- 必须覆盖：核心成果（表格）、工作详情、问题与风险、下周计划
- 核心成果必须用「数据 + 影响」描述，不用「做了优化」这类空话
- 下周计划按优先级排序，每条含「做什么」和「预期产出」

**取舍原则：** 数据说话 > 面面俱到。宁可少写两行，不写没有数据支撑的成果。

## 流程编排

单 Skill，串行执行：

1. 确认时间范围（默认本周一至周日）和汇报人信息
2. 收集工作素材（口述/列点/笔记/聊天记录均可）
3. 素材不足 → 追问三个核心问题
4. 整理分类 → 提炼核心成果 → 识别风险 → 生成下周计划
5. 输出完整周报
6. 用户确认后完成

## 协作规范

**节奏：** 一口气生成，不做中间确认。

**汇报：** 直接输出完整周报文档，不输出分析过程。

**追问原则（素材不足时）：**
- 每次只问一个问题
- 优先问：「这周最花精力的三件事是什么？」
- 其次问：「有没有遇到需要别人帮忙的问题？」
- 最后问：「下周最重要的一个目标是什么？」
- 不连问三个，逐个追问

**异常：**
- 用户说「这周没什么特别的」→ 追问「那日常工作中有没有什么变化？哪怕是小事」
- 用户给的素材杂乱 → 先按「完成的事项」「遇到的问题」「计划」三类归并，让用户确认后再写
- 用户没有明确部门/姓名 → 从 Memory 取，取不到用「待补充」占位

## 知识预设

**自动从 Memory 取用：**
- 姓名、部门
- 当前项目名称（如有）
- 常用周报格式偏好（如有）

**不从 Memory 取用：**
- 具体工作内容（每次不同，必须用户说）
- 数据和成果（必须用户提供，不编造）

## 交付规范

- 输出格式：完整 Markdown 文档
- 五个章节：报告周期与汇报人、核心成果（表格）、工作详情、问题与风险（表格）、下周计划
- 核心成果描述格式：「完成 X → 导致 Y」或「X 从 A 提升到 B」
- 问题与风险表格：问题描述 | 影响程度 | 解决状态 | 需支持
- 下周计划按高/中/低优先级排序
- 语气中立、专业，不加个人评价
- 不编造用户未提及的数据，数据缺失时标「待补充」
- 用户提供公司模板时，优先遵循公司模板
