# 会议纪要 Profile

## 适用场景

我说「整理会议纪要」「帮我写纪要」「把这段录音整理一下」「会议记录」时启用。

如果我说「帮我记一下」但没给任何素材，不启用，提示用户先提供素材。

如果只是「帮我看看这段录音说了什么」或「提取这段录音的要点」，不走完整纪要流程，直接提取要点即可。

## 技能依赖

**必选：** 会议纪要 Skill。

本场景不涉及其他 Skill。

## 目标标准

- 核心目标：重点突出、可直接发送给未参会者
- 必须覆盖：概要、讨论要点、行动项、决议
- 行动项必须有负责人 + 截止时间，缺一不可
- 区分「已达成共识」和「存在分歧」

**取舍原则：** 忠实 > 精致。不编造、不美化、不遗漏关键分歧。

## 流程编排

单 Skill，串行执行：

1. 确认素材类型和会议基本信息（主题、日期、参会人）
2. 素材过少 → 追问补齐
3. 素材充足 → 提炼概要 + 讨论要点 + 行动项 + 决议
4. 输出完整纪要
5. 用户确认后保存

## 协作规范

**节奏：** 一口气整理完成，不做中间确认。

**汇报：** 直接输出完整纪要文档，不输出「我正在分析」之类的过渡语。

**追问原则：**
- 素材中缺少参会人 → 追问一次
- 行动项缺负责人或截止时间 → 从素材中尽量推断，推断不出才追问
- 同一议题有多个方案未定论 → 如实记录，归入「待讨论」，不替用户选

**异常：**
- 素材太短/太乱 → 先按议题分类，给用户确认后再写纪要
- 大量闲聊内容 → 只提取与议题相关的讨论，忽略闲聊
- 素材涉及敏感信息 → 照常写入，但提醒「发送前请确认是否需要脱敏」

## 知识预设

- 会议主题、日期、参会人优先从用户输入获取
- 如果对话中已提及项目名，自动关联
- 不查外部资料，完全基于用户提供的素材
- 不从 Memory 自动取用会议相关信息（每次不同）

## 交付规范

- 输出格式：完整 Markdown 文档
- 四个必选章节：会议概要、讨论要点、行动项、会议决议
- 行动项用表格：事项 | 负责人 | 截止时间
- 讨论要点按议题分组，区分共识和分歧
- 使用第三人称：「张三提出」「李四建议」，不加整理者评价
- 语气中立、客观、专业
- 不编造素材中没有的结论或决议
