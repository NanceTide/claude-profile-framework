# 全局执行框架

## Profile 层

### Profile 是什么

Profile（业务场景配置层，Scenario Profile）是用户对**一个业务场景**执行方式的自然语言描述。它不是 YAML 配置，不是结构化字段，而是一张**自由便签**——用户用自己最舒服的方式写下「遇到这类任务时，我希望你怎么做」。

Profile 告诉 Agent：**这类业务在你这里要怎么理解、依赖哪些 Skill、怎么推进、怎么取材、怎么协作、怎么交付**。Agent 在执行时读取当前 Profile，加载 Profile 中声明的 Skill，并按 Profile 的指示决定如何串联步骤、如何汇报、如何从 Memory 取数。

```
Skill   = 企业/项目标准方法论（质量流程部门维护的公共只读资产）
Profile = 用户贴在具体业务场景上的执行契约（个性化打法，运行时最高优先级）
Memory  = 用户/组织的稳定事实，分两层：
          ├─ 全局 Memory：跨场景共享的稳定事实（用户角色、团队结构、跨场景偏好）
          └─ Profile Memory：场景专属的高复用事实（必须写在对应 Profile 文件内的「场景记忆」区）
```

**设计原则：**

- Profile 是**强制前置检查项**，不是可选增强项。**无论任务多简单，Agent 在开始执行前都必须先读取当前 Profile。**
- 若当前上下文尚未确定应使用哪个 Profile，Agent 也必须先完成 Profile 识别 / 候选列举 / 用户确认，而不是绕过 Profile 直接执行。
- Profile 是**场景级**的，不同业务场景彼此独立（如「写 Charter」与「提交请假」各有一份）。
- 用户**只改 Profile**，不修改 Skill；标准方法论由 Skill 恒定共享。
- Profile Memory 不是外部附属说明，也不是平台级临时缓存；**凡属本场景要持久化的记忆，都必须直接写入该 Profile 文件内的「场景记忆」区。**
- 系统解析时以**整块自然语言**为准；下文七个维度仅作写作引导，标题可删改、合并，也可完全不用标题。

---

### Profile 的形态

Profile 是一段自然语言文本，可长可短。以下**七个维度**作为建议骨架——在用户创建空白 Profile 时，系统可预置这些标题。用户拥有完全自由：

- 删除不需要的标题
- 修改标题措辞
- 合并或拆分段落
- 完全不用标题，写一段散文

**系统解析时以整块自然语言为准，标题仅用于帮助用户组织和理解内容，不产生结构依赖。**

---

### 七个维度

以下七个维度构成通用 Profile 架构，合起来描述一份完整的**场景执行契约**。

| 维度 | 概要 |
|------|------|
| 适用场景 | 何时启用、何时不启用、多场景如何选 |
| 技能依赖 | 本场景用到哪些 Skill、必选还是可选 |
| 目标标准 | 业务目标、完成定义、价值取舍 |
| 流程编排 | 多个 Skill 的执行顺序、并行与衔接 |
| 协作规范 | 暂停、汇报、异常与委托 |
| 知识预设 | 信源偏好、附件、Memory 取用 |
| 交付规范 | 体裁、格式、篇幅、禁忌措辞 |

---

#### 1. 适用场景

**回答：什么情况下启用这套打法？什么情况下明确不启用？多个场景同时沾边时怎么办？**

用户可能写：

> 我说「写 Charter」「立项报告」「产品规划」时启用。若我说「查点资料」「随便看看市场」，不走这套重流程，简单检索即可。若同时像「写规划」又像「写周报」，先列出候选项让我选。

边界：

- 允许模糊表达（「提到立项相关的事都算」）
- 允许否定触发（「除非我说随便看看，否则……」）
- 不强制精确关键词列表
- 多个 Profile 同时匹配时，主动列出候选项让用户选择
- 可选：相对优先级、互斥关系、时间或项目范围（如「仅 Q3 预算季」）

---

#### 2. 技能依赖

**回答：完成本场景会用到哪些 Skill？每个是必选还是可选？**

本维度回答 **「参与方清单」**，不回答先后顺序（顺序写在「流程编排」）。多 Skill 场景应**逐条列出**会用到的 Skill，便于 Agent 加载与核对。

用户可能写：

> **必选：** 市场洞察、竞争分析、财务预测、法务合规。  
> **可选：** 内部草稿可不做法务合规。  

单 Skill 示例：

> **必选：** 请假申请 Skill。本场景不涉及其他 Skill。

边界：

- 多 Skill 时建议分条写清**必选 / 可选**，避免 Agent 漏调或误调
- 允许用业务角色代称，系统映射到具体 Skill 名
- 与「流程编排」分工：**技能依赖 = 用谁；流程编排 = 何时用、按什么顺序、能否并行**
- Profile 可声明省略某 Skill；与 Skill 默认范围冲突时，**以 Profile 为准**

---

#### 3. 目标标准

**回答：这件事要达成什么业务结果？怎样算做完、算做好？在冲突时更倾向什么取舍？**

用户可能写：

> 目标是产出可立项的产品 Charter，能支撑评审会决策。必须覆盖：东南亚市场、云原生架构、Q3 里程碑。深度优先于速度；成本表述允许粗算，但关键假设必须写明。

边界：

- 写**业务目标**与**完成定义**（必须回答的问题、必须覆盖的主题/结论）
- 可写**软优先级**（深度 vs 速度、完整 vs 精简），非二元开关
- **不写**文档体裁、排版、页数（归「交付规范」）
- 组织级强制合规若写在 Profile 中，**以 Profile 为准**（高于 Skill 默认）

---

#### 4. 流程编排

**回答：多个 Skill 按什么顺序执行？哪些可以同时跑？上一步的输出如何交给下一步？条件不满足时如何分支？**

本维度回答 **「执行拓扑」**：在「技能依赖」已列出的 Skill 之上，规定**先后、并行、汇合与跳过**。流程编排中的每一步应尽量**点名 Skill**（或与技能依赖中的别名一致），便于 Agent 生成可执行的串联计划。

**推荐写法（多 Skill）：** 按阶段编号，标明并行与衔接。

> 1. **并行：** 市场洞察 + 竞争分析（两路同时跑，都完成后再进入下一步）。  
> 2. **串行：** 财务预测（输入：市场洞察 + 竞争分析的章节产出）。  
> 3. **串行：** 法务合规（输入：前述各章；可选，见技能依赖）。  
> 4. **汇合：** 将各章拼成完整 Charter（本步可不调用新 Skill，由 Agent 整合）。  
> **分支：** 无竞品数据时跳过竞争分析，其余顺序不变。

**单 Skill 示例：**

> 仅调用请假申请 Skill，一步填表提交，无并行、无汇合。

**编排用语（可在 Profile 中直接使用）：**

| 用语 | 含义 |
|------|------|
| 并行 / 同时跑 | 多个 Skill 可同时启动，全部完成后再进入下一阶段 |
| 串行 / 接着 / 再跑 | 前一 Skill 完成后再启动下一个 |
| 汇合 / 拼接 / 整合 | 将多个 Skill 的产出合并为最终交付物 |
| 跳过 | 满足条件时不调用某 Skill（须与技能依赖中的「可选」一致） |
| 输入：xxx 产出 | 标明上一步交付物作为下一步输入 |

边界：

- 多 Skill 场景**必须**在本维度体现顺序或并行；仅在技能依赖中列名而不写编排，视为编排不完整
- 允许只写一步（单 Skill 或「一步提交」）
- 与 Skill 各自内置流程冲突时，**以 Profile 编排为准**（调序、并行、跳过）
- 某步的暂停、汇报、异常处理写在「协作规范」，不在此重复
- 不涉及 Tool 底层调用细节（归各 Skill 内部逻辑）

---

#### 5. 协作规范

**回答：执行过程中何时需要你介入、何时汇报、卡壳或缺料时怎么办？**

本维度统摄人机协作，建议分三块书写（可合并成一段，不必分标题）：

**检查点与节奏** — 何时暂停、审批、确认；哪些步骤可一口气跑完；是否委托他人或其他 Agent。

> 市场洞察和竞争分析一口气写完。财务预测写完必须暂停，我确认数字后再继续。法务章交给法务助理，他审完你再拼回来。全文拼接前先看大纲。

**汇报** — 何时同步进度、摘要还是全文、超时如何提醒。

> 每章完发摘要，别发全文。财务章写完贴全文。某章超过十分钟没结果，告诉我卡在哪。

**异常与韧性** — 缺料、理解偏差、委托超时、系统报错时的默认策略。

> 查不到数据先用「待补充」占位，全文跑完再标红。法务助理半小时没回就跳过该章继续拼。听不懂就直接说，别硬猜。

边界：

- 允许全局或分步骤策略
- Tool / 外部 Agent 的协作偏好可写在本维度或「知识预设」
- **暂停、汇报、跳过**等以 Profile 为准，覆盖 Skill 默认节奏

---

#### 6. 知识预设

**回答：查什么料、不查什么料、参考哪些附件？执行时从 Memory 或对话中取哪些预设信息？哪些事实值得持久化？**

本维度同时承担**记忆策略声明**，但范围必须严格收敛：不是所有"自动带上""无需每次重复填写"的内容都应该写入 Profile Memory。只有**长期稳定、跨多次执行仍成立、且在本场景高频复用**的事实，才应持久化。

用户可能写：

> 市场洞察优先查 xx 网站行业月报，别用 Gartner。竞争分析必须覆盖 A 公司最近发布会。财务去 ERP 拉，没权限就标黄「待补充」，别编造。参考我上个月传的《Q1 复盘》。自动带上研发部和直属领导李总；请假自动填工号；跟项目有关的活，优先读取当前对话里的项目名，别让我重复报。

**Memory 桥梁：** 约定执行**本场景**时，必须从全局 Memory 或当前对话取哪些信息、填到哪些环节；同时声明哪些场景专属事实应持久化到 Profile Memory。这里的 Profile Memory **必须落在当前 Profile 文件本身**：凡需沉淀的场景事实，都写入该文件内的「场景记忆」区，而不是写到别处。

边界：

- 允许来源白名单 / 黑名单
- 允许附件与历史材料引用
- 不强制数据源已系统接入，允许自由文本描述
- 查不到时的处理可与「协作规范」中异常策略呼应（占位 / 追问 / 标黄）
- 允许声明**读取来源优先级**：先看当前对话，再看 Profile Memory，最后追问用户
- "自动带上"不等于"自动持久化"；只有符合持久化标准的事实才写入 Profile Memory
- 执行策略、追问规则、数据源偏好、当前任务上下文，不得写入 Memory 范围
- Profile 在本维度应尽量给出**窄白名单**，明确"哪些能记"与"哪些不能记"，避免笼统写成"项目相关信息""学习上下文""会议信息"

---

#### 7. 交付规范

**回答：交付物是什么体裁、什么结构、什么语气？有哪些格式与措辞要求？**

用户可能写：

> 整体像内部汇报，别像学术论文。先列摘要，最后可补细节。每章不超过两页。数字用「约 xx 万」，别精确到小数点后两位。能画图就直接出图。不要出现「亏损」，改用「投入期」。

边界：

- 定性（简洁、正式）与定量（篇幅、精度）均可
- 格式（Markdown、PPT 大纲等）与禁忌措辞均写在此
- **验收性**要求（必须覆盖的主题、结论）写在「目标标准」，此处专注**表达与形态**
- 与 Skill 输出格式冲突时，**以 Profile 为准**

### Profile 与 Skill 的边界

```
Skill 负责（组织标准，只读共享）：          Profile 负责（用户场景契约，可读写）：
├─ 标准方法论与通用规范                     ├─ 适用场景（何时启用）
├─ 单 Skill 内的 Tool 与执行逻辑            ├─ 技能依赖（用哪些 Skill）
├─ 合规检查项与质量门禁                     ├─ 目标标准（怎样算做成）
├─ 默认执行逻辑与异常分类                   ├─ 流程编排（步骤与并行）
└─ 组织级必填与审计要求                     ├─ 协作规范（暂停、汇报、异常）
                                            ├─ 知识预设（来源、Memory 桥接 + 记忆策略）
                                            ├─ 交付规范（体裁、格式、禁忌）
                                            └─ 场景记忆（Profile Memory，高复用事实）
```

**优先级规则：Profile 具有最高权限，且读取当前 Profile 是每次执行的硬前置条件。**

Agent **不得**因为任务简单、看似通用、时间紧、已有经验或能直接回答，就跳过当前 Profile 的读取。即使是简单问答、轻量改写、快速检索、一次性小任务，也必须先读取当前 Profile，再决定如何执行。

当 Profile 与 Skill 冲突时，**以 Profile 为准**，包括但不限于：依赖哪些 Skill、执行顺序与是否并行、是否暂停等待确认、数据源选择、交付格式与措辞、是否跳过某步骤、预设信息注入范围。Skill 提供 baseline；Profile 是用户在该业务场景下的最终意志。

仅以下情况不受 Profile 覆盖（由平台或法律强制）：

- 模型与平台安全红线
- 显式违法或越权操作请求

Skill 升级时，系统应提示用户复核相关 Profile（尤其「技能依赖」「流程编排」），确保仍与现行 Skill 兼容。

### Profile 与 Memory

Memory 分为两层：**全局 Memory** 和 **Profile Memory**。

```
全局 Memory（跨场景共享）              Profile Memory（场景专属）
├─ 用户角色、团队结构、汇报关系          ├─ 该场景积累的领域事实
├─ 跨场景通用偏好与反馈                 ├─ 该场景下复用率极高的业务数据
└─ 全局 reference（系统入口、面板地址）   └─ Profile 声明值得持久化的事实
```

#### Profile Memory 是什么

Profile Memory 是**该场景下复用率极高的稳定事实**。不是所有执行中产生的事实都值得存；只有那些**跨多次执行仍成立、几乎每次都会用到、且重新询问成本高于误记风险**的事实，才应该写入。并且，这些事实的落点必须是**对应 Profile 文件内的「场景记忆」区**。

Profile 不只是执行契约，它同时是一个**记忆策略**——声明哪些事实值得持久化。Profile 的自然语言描述中，涉及"自动带上""无需每次重复填写"的表述时，Agent 必须先判断该事实是否满足持久化标准；只有满足时，才同时完成下面两件事，并把该事实写入当前 Profile 文件内的「场景记忆」区：

1. **读取指令** — 执行时从 Memory 取该事实，避免重复询问
2. **写入指令** — 该事实值得持久化，供未来执行复用

#### Profile Memory 的准入标准

一条事实只有在**同时满足**以下条件时，才允许写入 Profile Memory：

1. **场景专属**：主要只在该 Profile 对应的场景里使用
2. **跨次稳定**：到下一次、下下次执行时，大概率仍然成立
3. **高频复用**：该场景下多数执行都会用到
4. **低歧义**：字段含义清楚，不是模糊的大类描述

只满足其中一部分，不得写入。

反过来，凡是属于以下任一类，默认**禁止**写入 Profile Memory：

- 当次任务上下文：本次项目名、本次会议主题、本周工作、本次请假日期
- 时效性强的数据：剩余年假、临时风险、当前进度、一次性审批链
- 执行策略：先查哪里、是否追问、如何暂停、怎样汇报
- 来源规则：优先看附件、不要查外部、从当前对话提取
- 模糊集合：项目相关信息、学习上下文、会议信息、用户最近提到的内容

#### 什么该存，什么不该存

| | 该存入 Profile Memory | 不该存入 Profile Memory |
|---|---|---|
| 特征 | 场景专属、长期稳定、高频复用 | 当次上下文、强时效、临时变化 |
| 生命周期 | 长期有效或缓慢变化 | 一次性或短周期即过期 |
| 举例 | 工号、部门、直属领导、常用审批人、固定输出偏好 | 本次请假理由、本次请假日期、本周具体工作内容、本次会议决议、当前项目名 |

**判断标准：** 必须同时问四个问题：

1. 下次执行同一场景时，它大概率还成立吗？
2. 下次执行时，它大概率还会被用到吗？
3. 它是一个清晰字段，而不是一类模糊上下文吗？
4. 写错它的风险，是否低于每次重新追问一次的成本？

四个问题都偏向"是"，才存；否则不存。

#### Profile Memory 与全局 Memory 的边界

- **全局 Memory**：跨场景共享的事实（用户身份、团队结构、通用偏好）。多个 Profile 都会用到。
- **Profile Memory**：仅在该场景下高频复用、且跨次稳定的事实。不是所有场景资料、业务数据、项目背景都能进入。

Profile「知识预设」维度同时声明两种来源：全局的写"从全局 Memory 取"，场景特有且符合准入标准的写入当前 Profile 文件内的 Profile Memory；仅对当前任务有效的信息，保留在当前对话，不持久化。

#### Memory 的生命周期

事实会过期。Profile Memory 需要定期清理：

- **长期有效**（工号）：写入后极少更新
- **缓慢变化**（直属领导）：写入后偶尔更新，变化时重写
- **频繁变化**（剩余年假）：默认不写入；若业务必须依赖，也只可视为待核验字段，不作为稳定事实持久化
- **一次性**（请假理由）：不写入 Memory

Agent 在执行时应判断事实的时效性，过期的事实应更新或移除。

#### Profile 生成时的 Memory 写作要求

生成新的 Profile 时，Agent 在 Memory 相关段落必须遵守以下写法：

- 优先写成**白名单**："自动取用以下事实：……"
- 同时写出**排除项**："以下信息不要写入 Memory：……"
- 能写字段名就不要写笼统类别；写"工号、部门、直属领导"，不要写"个人信息"或"项目信息"
- 当前对话里可临时提取的内容，应写成"从当前对话取"，而不是写入 Memory
- 若某字段变化频繁但又常用，应标为"执行时核验"，而不是直接持久化
- 生成出的 Profile 文件中，必须显式写明：本场景需要持久化的 Memory 存放在**该 Profile 文件内的「场景记忆」区**
- 若 Profile 内声明了可持久化字段，文件中必须有对应的「场景记忆」区承接这些字段，不能只写规则不写落点

如果拿不准某条信息该不该持久化，默认**不写入 Memory**，只在当次对话中使用。

### 多 Skill 串联：技能依赖与流程编排如何配合

复杂业务（如写 Charter）往往**一个 Profile 对应多个 Skill**。体现方式如下。

```
技能依赖（参与方清单）          流程编排（执行拓扑）
        │                              │
        ▼                              ▼
  列出会用到的 Skill          规定顺序、并行、汇合、跳过
        │                              │
        └──────────┬───────────────────┘
                   ▼
            合并为执行计划
                   │
                   ▼
     按编排依次/并行调用各 Skill，并按协作规范暂停与汇报
```

**写作分工（避免只写其一）：**

| 要写的内容 | 写在哪一维 | 示例 |
|------------|------------|------|
| 会用到哪些 Skill | 技能依赖 | 必选：市场洞察、竞争分析、财务预测 |
| 先跑谁、后跑谁 | 流程编排 | 先并行市场洞察与竞争分析，再财务预测 |
| 能否同时跑 | 流程编排 | 市场洞察与竞争分析并行 |
| 上一步结果给谁用 | 流程编排 | 财务预测输入：前两章产出 |
| 最终如何成稿 | 流程编排 | 汇合：拼接为完整文档 |
| 某 Skill 是否可不跑 | 技能依赖 + 流程编排 | 可选：法务合规；无竞品则跳过竞争分析 |
| 某步完成后是否暂停 | 协作规范 | 财务预测完成后暂停待确认 |

**完整示例（写 Charter，节选）：**

技能依赖：

> 必选：市场洞察、竞争分析、财务预测、法务合规。可选：内部草稿可跳过法务合规。

流程编排：

> 1. 并行执行市场洞察、竞争分析。2. 串行执行财务预测（输入前两章）。3. 串行执行法务合规。4. 汇合拼接全文。无竞品数据时跳过竞争分析。

协作规范（与编排交叉时）：

> 步骤 1 一口气跑完；步骤 2 完成后暂停待我确认数字；步骤 3 可委托法务助理。

Agent 解析时：先根据技能依赖加载全部必选 Skill，再按流程编排生成**带顺序与依赖关系的执行计划**；Profile 优先级高于各 Skill 自带默认顺序。

### Profile 使用流程

```
用户发起任务
    │
    ▼
1. 解析用户输入，理解业务意图
    │
    ▼
2. 匹配 Profile（按「适用场景」）
   有 → 加载该场景 Profile
   无 → 不假设场景偏好，进入步骤 3
   多个命中 → 列出候选项让用户选择
    │
    ▼
3. 按 Profile「技能依赖」解析参与 Skill 列表（必选/可选），加载对应 Skill
   无 Profile 时 → 仅按 task_type 匹配 Skill
    │
    ▼
4. 按 Profile「流程编排」生成执行计划：顺序、并行、汇合、跳过条件；
   与技能依赖交叉校验（编排中提到的 Skill 须在依赖中已声明）
   再以 Profile 为最高优先级覆盖各 Skill 默认流程
    │
    ▼
5. 信息补全与 Memory 读取：
   a. 按 Profile「知识预设」从全局 Memory 取跨场景事实（用户角色、团队等）
   b. 从 Profile Memory 取场景专属事实（该场景积累的领域数据）
   c. 从当前对话取上下文信息
   d. Skill 必填项若 Profile 未禁止且仍缺失，可追问用户
    │
    ▼
6. 执行：按计划调用各 Skill；逐步推进时遵循「协作规范」
    │
    ▼
7. 交付：满足 Profile「目标标准」与「交付规范」
    │
    ▼
8. Memory 写入与清理：
   a. 按 Profile「知识预设」的持久化策略，将高复用事实写入 Profile Memory
   b. 用户反馈（纠正或确认）写入全局 Memory
   c. 过期的 Profile Memory 事实更新或移除
```

### Skill 与 Profile 查找路径

按以下路径查找；**同一冲突下，Profile 优先于 Skill**。

| 优先级 | 位置 | 范围 | 说明 |
|--------|------|------|------|
| 1 | `~/.claude/skill-definitions/` | 全局 Skill | 所有项目可见 |
| 2 | `~/.claude/profiles/` | 全局 Profile | 所有项目可见 |
| 3 | `~/.claude/skills/` | Claude Code 原生 Skill | 仅当前项目 |

**Memory 存储位置：**

| 类型 | 位置 | 说明 |
|------|------|------|
| 全局 Memory | `~/.claude/projects/-Users-nancetide/memory/` | 跨场景共享的稳定事实 |
| Profile Memory | 各 Profile 文件内的「场景记忆」区 | 场景专属的高复用事实，随 Profile 文件一起管理 |

**同名冲突时：用户级覆盖项目级（用户 > 项目）。**

若仅有 Profile、无 Skill，Profile 直接指导通用 Agent 的执行风格、编排与交付；「技能依赖」中声明的 Skill 名可用于提示缺失能力。若仅有 Skill、无 Profile，按 Skill Template 默认执行，不影响正常使用。
