# Skill 名称

写周报



## Skill 目标

根据用户提供的零散工作记录（口述、笔记、任务列表等），自动生成一份结构清晰、重点突出、符合企业规范的周报。



## 何时使用

- 用户需要撰写周报时

- 用户要求“帮我写周报”

- 用户给出本周工作要点，希望整理成正式周报格式


## 输入

用户提供以下任意一种或多种形式的信息：

- 口述：本周做了哪些事

- 文字列点：如“1. 完成A项目方案；2. 修复B系统bug……”

- 每日工作笔记截图或文字

- 聊天记录、会议记录、任务清单

- 若有特定格式要求（如公司固定模板），用户可一并提供



## 输出

生成一份Markdown格式的周报，包含以下固定章节：

- 报告周期与汇报人信息

- 一、本周核心成果（表格形式）

- 二、本周工作详情（已完成 + 进行中）

- 三、问题与风险（表格形式）

- 四、下周工作计划（优先级列表）

- 五、备注（可选）



## 核心流程

1. **确认时间范围**：询问用户周报的起止时间（默认本周一至周日），以及汇报人姓名/部门。

2. **收集工作信息**：

  - 引导用户按三个维度补充信息：完成了什么？结果/产出是什么？遇到什么问题？

  - 如果用户只给了任务名称，追问“这个任务的结果是什么？有没有数据或影响？”

3. **整理分类**：

  - 将工作事项归入“已完成”“进行中”

  - 识别其中可量化的成果（如“完成3篇文档”“修复5个bug”），放入“核心成果”表格

4. **识别风险**：

  - 如果用户提到困难、延期、依赖他人，自动提炼到“问题与风险”表格

  - 若用户说“没问题”，则该项留空或写“无”

5.**生成下周计划**：

  - 根据本周未完成事项 + 用户明确提及的下周任务，生成3条左右的优先级计划

  - 每条计划需包含“做什么”和“预期产出”

6. **输出并确认**：将生成的周报呈现给用户，询问是否需要修改或补充。



## 示例或模版

```markdown

# 周报 - 张三（产品部）

报告周期：2026年5月20日 — 5月26日（第21周）

汇报日期：2026年5月27日



一、本周核心成果

| 工作事项 | 产出/影响 |

|----------|-----------|

| 完成V2.0需求评审 | 通过评审，进入开发 |

| 修复线上3个高优先级bug | 系统崩溃率下降40% |



二、本周工作详情

**已完成：**

- V2.0需求文档终稿，已同步给研发和测试

- 组织两次跨部门对齐会，输出会议纪要



**进行中：**

- 用户手册更新：进度60%，预计本周五完成



三、问题与风险

| 问题描述 | 影响程度 | 解决状态 | 需支持 |

|----------|----------|----------|--------|

| 测试环境不稳定 | 高 | 跟进中 | 请运维协助排查 |



四、下周工作计划

1. 完成用户手册更新并发布

2. 跟进V2.0开发进度，协调资源

3. 整理Q2产品数据复盘



五、备注

无



##执行偏好或规则

优先使用“数据+影响”的描述方式，避免空话（如“做了优化” → “响应时间从3秒降到1秒”）



必须包含“问题与风险”章节，即使填写“无”



禁止直接复制用户原话而不加整理



不要编造用户未提及的数据；若数据缺失，主动询问



推荐将下周计划按优先级排序（高/中/低）



##Memory 使用

如果本次会话中用户已经提供过姓名、部门、常用项目名称，后续生成周报时可以直接复用，无需重复询问。



##错误或异常处理

用户提供的信息过少（如只说“做了些事”）：主动追问三个问题：



最花精力做的三件事是什么？

有没有遇到需要别人帮忙的问题？

下周最重要的一个目标是什么？

用户给的素材杂乱：先按“完成的事项”“遇到的问题”“计划”三类进行归并，让用户确认是否正确



用户没有明确的部门/姓名：先使用“【待补充】”占位，并在输出末尾提醒用户自行填写



##其他要求

生成的周报应保持中立、专业的语气，不要添加个人评价（如“我真的很努力”）



如果用户提供公司特定模板格式，优先遵循用户模板，而不是本Skill的默认模板
