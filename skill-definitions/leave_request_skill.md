# 提交请假申请 Skill

## Skill 目标

帮助用户完成企业内部请假申请，包括识别请假意图、抽取请假信息、补全必要字段、生成规范化申请文本或系统表单内容，并在用户确认后协助提交或转交。

本 Skill 不替用户决定请假事项，不编造日期、类型、时长、原因或审批人。

## 何时使用

当用户表达以下或类似意图时使用：

- 我要请假
- 帮我提交请假申请
- 明天有事想请假
- 下周一请年假
- 我身体不舒服，想请病假
- 帮我给领导发请假申请

当用户只是咨询请假制度、询问剩余年假、询问流程，但尚未要求生成或提交申请时，先解释或确认，不直接进入提交流程。

## 输入

### 用户显式输入

优先从当前对话提取：

- 请假人姓名
- 所属部门
- 工号或员工编号
- 请假类型：年假、事假、病假、调休、婚假、产假、丧假等
- 开始时间
- 结束时间
- 请假时长
- 请假事由
- 直属领导或审批人
- 抄送人
- 工作交接安排
- 输出形式：正式文本、邮件、OA 表单、即时消息
- 是否需要提交、发送或转交

### 可从 Memory 读取的信息

- 用户姓名
- 工号
- 所属部门
- 岗位名称
- 直属领导
- 常用审批人
- 常用抄送人
- 常用工作交接人
- 企业请假格式偏好

### 不得擅自编造的信息

以下信息缺失时必须确认，不得编造：

- 请假具体日期和时间
- 请假类型
- 请假时长
- 请假事由
- 审批人
- 是否提交
- 是否抄送他人

## 输出

根据用户需求输出以下一种或多种结果。

### 请假申请文本

适用于发送给领导、HR 或团队群，包含：

- 称呼
- 请假人基本信息
- 请假类型
- 请假时间
- 请假时长
- 请假事由
- 工作交接说明
- 审批请求
- 署名和日期

### 请假表单字段

适用于 OA、HR 系统、飞书、钉钉、企业微信等系统：

```yaml
请假人:
部门:
工号:
请假类型:
开始时间:
结束时间:
请假时长:
请假事由:
审批人:
抄送人:
工作交接:
备注:
```

### 提交前确认

任何会触达审批人或系统的动作前，必须展示摘要并等待用户确认：

```text
以下是我整理的请假申请信息，请确认是否提交：
……
请回复「确认提交」或指出需要修改的内容。
```

## 核心流程

1. **识别意图。** 判断用户是咨询制度，还是要生成/提交请假申请。
2. **抽取信息。** 从对话中抽取请假类型、时间、时长、事由、审批人、输出形式等字段。
3. **读取 Memory。** 读取姓名、部门、工号、直属领导、常用审批人等稳定信息。
4. **检查完整性。** 提交前至少确认请假人、部门、请假类型、开始时间、结束时间、时长、事由、审批人。
5. **生成草稿。** 按用户需要生成申请文本、表单字段或即时消息。
6. **提交前确认。** 若用户要求发送、提交或转交，必须先展示最终内容并等待确认。
7. **更新 Memory。** 若用户提供或更正了长期复用信息，则更新这些稳定事实；一次性请假内容不应写入 Memory。

## 质量约束

- 不编造具体日期、请假类型、时长、原因、审批人。
- 日期含糊时必须转成明确日期，无法确认则追问。
- 草稿可以用「待补充」占位；提交前不得保留关键字段空缺。
- 请假理由保持真实、简洁、不过度披露隐私。
- 病假、婚假、产假、丧假等可能涉及证明材料时，提醒用户按公司制度补充附件。
- 直接提交、发送、转交前必须得到用户明确确认。

## 异常处理

| 情况 | 处理方式 |
|------|---------|
| 用户只说「我要请假」 | 追问请假类型、时间、时长、事由 |
| 缺少工号/部门/领导 | 先从 Memory 取；取不到再问 |
| 日期是「明天」「下周一」 | 结合当前日期换算为具体日期，并向用户确认 |
| 请假理由过于私人 | 帮用户改写为简洁中性表述 |
| 用户要求直接提交 | 展示最终内容，等待「确认提交」 |
| 剩余假期不确定 | 标「待核验」，不推断余额 |

## Memory 使用

请假场景下，可长期复用的信息限于稳定身份与常用审批链。

适合写入 Memory 的典型信息：

- 工号
- 部门
- 岗位
- 直属领导
- 常用审批人
- 常用抄送人
- 常用工作交接人
- 常用请假系统入口

不应写入 Memory 的信息：

- 本次请假事由
- 本次请假日期
- 本次请假时长
- 临时抄送人
- 一次性备注
