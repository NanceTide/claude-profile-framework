# 提交请假申请_skill

## Skill 目标

本 Skill 用于帮助用户完成企业内部“提交请假申请”相关任务，包括识别请假意图、收集请假信息、补全必要字段、生成规范化请假申请内容，并在用户确认后协助进入提交或转交流程。

本 Skill 的目标不是替用户擅自决定请假事项，而是按照企业常见请假流程，帮助用户形成完整、准确、合规、可提交的请假申请。

---

## 何时使用

当用户表达以下或类似意图时，应使用本 Skill：

- “我要请假”
- “帮我提交请假申请”
- “明天有事想请假”
- “帮我写一份请假说明”
- “下周一请年假”
- “我身体不舒服，想请病假”
- “帮我给领导发请假申请”

当用户只是咨询假期制度、询问剩余年假、询问请假流程，但尚未明确要生成或提交申请时，可先进行解释或信息确认，不应直接进入提交流程。

---

## 输入

### 用户显式输入

执行本 Skill 时，优先从用户当前对话中提取以下信息：

- 请假人姓名
- 所属部门
- 工号或员工编号
- 请假类型，例如年假、事假、病假、调休、婚假、产假、丧假等
- 请假开始时间
- 请假结束时间
- 请假时长
- 请假事由
- 直属领导或审批人
- 是否需要抄送相关人员
- 是否需要生成正式文本、邮件、系统表单内容或即时消息内容
- 是否需要直接提交

### 可从 Memory 中读取的信息

在用户允许或系统配置允许的情况下，可从 Memory 中读取以下稳定信息：

- 用户姓名
- 用户工号
- 所属部门
- 岗位名称
- 直属领导
- 常用审批人
- 常用抄送人
- 用户常用请假事由表达偏好
- 用户所在企业或团队的请假格式偏好


### 草稿与提交的字段边界

- 当用户只要求“写个请假申请”“生成草稿”，且已提供请假类型、请假时间或时长、请假事由等核心请假信息时，可以先生成草稿；但缺失的请假人、部门、工号、审批人、抄送人、工作交接等字段必须以“待补充/待确认”标注，或明确说明需要从 Memory 获取或向用户确认。
- 当用户要求“提交”“发送”“发给某人”“转交”或任何会触达审批人/系统的动作时，必须先展示申请摘要和最终文本/字段，并请求用户确认，不得直接执行。
- 当用户仅表达“我要请假”“帮我请假”等泛化意图，且缺少请假类型、开始时间、结束时间或时长、事由、审批人等核心信息时，不得生成看似完整的申请，应暂停并追问。
- 审批人、部门、工号等组织信息可从 Memory 获取；若 Memory 不可用或存在多个候选，必须说明并向用户确认。
### 不得擅自编造的信息

以下信息如缺失，必须向用户确认，不得自行编造：

- 请假具体日期和时间
- 请假类型
- 请假时长
- 请假事由
- 审批人
- 是否提交
- 是否抄送他人

## 输出

根据用户需求，输出以下一种或多种结果：

### 1. 请假申请文本

适用于需要发送给领导、HR 或团队群的场景。输出应包含：

- 称呼
- 请假人基本信息
- 请假类型
- 请假时间
- 请假时长
- 请假事由
- 工作交接说明
- 审批请求
- 署名和日期

### 2. 请假表单字段

适用于企业 OA、HR 系统、飞书、钉钉、企业微信等请假系统。输出应为结构化字段，例如：

```yaml
请假人:
部门:
工号:
请假类型:
开始时间:
结束时间:
请假时长:
请假事由:
审批人:
抄送人:
备注:
```

### 3. 确认提示

在提交前必须向用户展示待提交内容，并请求确认：

```text
以下是我整理的请假申请信息，请确认是否提交：
……
请回复“确认提交”或指出需要修改的内容。
```

如果运行环境需要结构化输出，应按以下格式返回：

```yaml
  status: "success_or_partial_or_failed_or_paused"

    - "等待用户确认"

  inputs_used: []
  memory_used: []

  warnings: []
  errors: []

  profile_name: ""
  overrides: []
```

## 核心流程

### 第一步：识别请假意图

判断用户是否明确要办理请假申请。

如果用户只是询问制度，例如“病假怎么请”“年假能请几天”，则先回答制度或流程问题，不进入申请生成流程。

如果用户明确表达请假需求，则进入下一步。

---

### 第二步：抽取已知信息

从用户当前输入中抽取请假申请所需字段，包括：

- 请假类型
- 请假时间
- 请假时长
- 请假事由
- 审批对象
- 输出形式

示例：

用户输入：

```text
我下周一上午想请半天事假，家里有点事，帮我给领导写一下。
```

可抽取为：

```yaml
请假类型: 事假
开始时间: 下周一上午
结束时间: 下周一上午
请假时长: 半天
请假事由: 家中事务
输出形式: 给领导的请假文本
```

---

### 第三步：读取 Memory 中的稳定信息

如当前输入缺少姓名、部门、工号、直属领导等基础信息，可根据 Profile 或系统配置从 Memory 中读取。

读取 Memory 时应遵守以下规则：

- 只读取本次请假申请必要的信息；
- 不得读取与请假无关的私人信息；
- 使用 Memory 后，应在结构化输出中的 `context.memory_used` 中记录；
- 如果 Memory 中存在多个可能值，应向用户确认；
- 如果 Memory 缺失，不得编造，应询问用户。

---

### 第四步：检查关键字段是否完整

提交请假申请前，至少应确认以下字段：

- 请假人
- 部门或岗位
- 请假类型
- 开始时间
- 结束时间
- 请假时长
- 请假事由
- 审批人或接收对象

如果缺少关键字段，应暂停并询问用户。

示例：

```text
还缺少两个信息：请假类型和请假结束时间。请问这次是事假、年假、病假还是调休？请假到哪一天或几点结束？
```

---


### 用户要求“随便写”或原因不明确

如果用户要求“原因随便写”“你帮我编一个理由”“写得严重一点”等，必须拒绝编造具体事实，不得代替用户虚构病情、家庭事务、证明材料或其他不真实原因。应请用户提供真实事由，或建议使用不虚构事实的中性表达，例如“因个人事务需请假”或“因身体不适需请假”。如果用户仍不提供真实或中性事由，应暂停生成正式申请。
### 第五步：规范化请假事由

请假事由应正式、简洁、中性，不应过度暴露个人隐私。

示例规则：

- 用户说“家里有点事”，可规范为“因个人事务需请假”；
- 用户说“身体不舒服”，可规范为“因身体不适需请假”；
- 用户说“去医院检查”，可规范为“因就医检查需请假”；
- 不主动补充病情、家庭矛盾、具体隐私细节；
- 不夸大、不虚构、不编造证明材料。

---

### 第六步：生成请假申请草稿

根据用户指定的输出形式生成内容。

#### 正式文本模板

```text
尊敬的{审批人/领导}：

您好！本人{姓名}，系{部门}{岗位/员工}，因{请假事由}，拟申请{请假类型}，请假时间为{开始时间}至{结束时间}，共计{请假时长}。

请假期间，我将提前做好相关工作安排，并与{交接人/相关同事}做好工作交接，确保不影响工作正常推进。

恳请批准。

申请人：{姓名}
{日期}
```

#### 简洁消息模板

```text
{审批人/领导}您好，我因{请假事由}，需于{请假时间}请{请假类型}{请假时长}。相关工作我会提前安排并做好交接，烦请您审批，谢谢。
```

#### 表单模板

```yaml
请假人: "{姓名}"
部门: "{部门}"
工号: "{工号}"
请假类型: "{请假类型}"
开始时间: "{开始时间}"
结束时间: "{结束时间}"
请假时长: "{请假时长}"
请假事由: "{请假事由}"
审批人: "{审批人}"
抄送人: "{抄送人}"
工作交接: "{交接说明}"
备注: "{备注}"
```

---

### 第七步：提交前确认

除非用户明确只要求“生成草稿”，否则在任何提交、发送、转交前，必须请求用户确认。

确认提示应包括：

- 请假类型
- 请假时间
- 请假时长
- 请假事由
- 审批人
- 最终文本或表单字段

不得在用户确认前擅自提交、发送或通知他人。

---

### 第八步：执行后反馈

如果用户确认提交，并且运行环境具备提交能力，则执行提交，并反馈结果。

如果运行环境不具备提交能力，则输出可复制文本或表单字段，并说明需要用户手动提交。

示例：

```text
我已为你整理好请假申请内容。当前环境无法直接提交 OA，请复制以下内容到请假系统中提交。
```

## 执行偏好或规则

### 必须遵守

- 对于只要求生成草稿的场景，允许在核心请假信息充分时先生成草稿，并把缺失的组织字段标为待确认；但不得提交或发送。
- 对于提交、发送、转交场景，缺少任何提交必需字段时必须暂停确认，且即使字段完整也必须先请求用户最终确认。
- 当用户要求“随便写原因”或暗示编造理由时，必须拒绝编造，并要求真实或中性事由。

- 必须确认请假开始时间和结束时间；
- 必须确认请假类型；
- 必须确认请假事由；
- 必须在提交或发送前请求用户确认；
- 必须使用正式、简洁、符合企业场景的语言；
- 必须保护用户隐私，不主动扩写敏感个人原因；
- 必须对缺失字段进行追问；
- 必须区分“生成草稿”和“直接提交”。

### 禁止事项

- 禁止编造用户部门、工号、领导、请假时间；
- 禁止擅自提交未经用户确认的申请；
- 禁止编造病情、证明材料或虚假原因；
- 禁止将请假事由写得过于私人化或情绪化；
- 禁止使用不符合企业场景的口语化表达；
- 禁止在没有权限或工具能力时声称已经提交成功。

### 优先规则

- 优先使用用户明确提供的信息；
- 用户未提供但 Memory 中存在稳定信息时，可按 Profile 指示使用；
- 请假事由优先使用中性表达；
- 如果用户指定格式，以用户格式为准；
- 如果企业有固定模板，以企业模板为准。

## Memory 使用

### 可读取的 Memory

- 姓名
- 工号
- 部门
- 岗位
- 直属领导
- 常用审批人
- 常用抄送人
- 常用请假格式偏好

### Memory 使用规则

- 仅在请假申请需要时读取；
- 只读取与本次申请直接相关的信息；
- 使用 Memory 前后应保持可解释；
- Memory 信息与用户当前输入冲突时，以用户当前输入为准，并提示用户确认；
- Memory 中缺少关键信息时，应向用户询问；
- 不应将敏感或无关 Memory 注入请假申请。

### 示例

如果 Memory 中已有：

```yaml
姓名: 张三
部门: 产品部
工号: A00123
直属领导: 李经理
```

用户输入：

```text
我明天上午请半天事假，家里有点事。
```

则可生成草稿，并在提交前提示用户确认审批人是否为李经理。

## 错误或异常处理

### 信息缺失

如果缺少关键字段，返回 `paused` 状态并追问。

示例：

```yaml
metadata:
  status: "paused"
  requires_confirmation: true
errors:
  has_errors: true
  warnings:
    - "缺少请假开始时间"
    - "缺少请假类型"
```

### 时间不明确

如果用户使用“明天”“下周一”等相对时间，应结合当前日期推断；如无法确定，应询问确认。

示例：

```text
你说的“下周一”是指 2026 年 X 月 X 日吗？请确认。
```

### 时间冲突

如果结束时间早于开始时间，应提示用户修正。

```text
请假结束时间早于开始时间，请重新确认请假起止时间。
```

### 审批人不明确

如果审批人缺失，应询问用户，或根据 Memory 建议但不能擅自最终确认。

```text
我在记忆中看到你的直属领导可能是李经理，是否将李经理作为审批人？
```

### 工具失败

如果 OA、邮件、IM 等提交工具失败，应说明失败原因，并提供可复制的备用文本。

```text
当前无法连接请假系统，我已为你生成可复制的请假申请内容，你可以手动提交。
```

### 权限不足

如果当前环境无权限提交申请，不得声称已提交，应明确说明只能生成草稿或字段。

---

## 其他要求

- 默认输出中文；
- 默认语气正式、简洁、礼貌；
- 默认不暴露过多个人隐私；
- 默认先生成草稿，再请求确认；
- 若用户要求英文请假邮件，可输出英文版本；
- 若用户要求发送给领导，应生成适合即时消息或邮件的版本；
- 若用户要求提交系统，应优先输出结构化字段；
- 若请假涉及长期病假、产假、婚假、丧假等特殊假种，应提醒用户可能需要上传证明材料，但不得编造材料；
- 若请假影响重要会议、值班、交付节点，应建议用户补充工作交接安排。
