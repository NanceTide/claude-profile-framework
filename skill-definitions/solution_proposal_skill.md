# 解决方案提案 Skill

## Skill 目标

围绕客户需求、业务问题和交付目标，生成可用于售前提案、项目方案说明或客户汇报的结构化方案正文。

本 Skill 负责把零散需求整理成有逻辑的解决方案文档，突出问题理解、方案设计、实施路径、边界与建议，不替用户虚构客户背景、交付能力、成功案例或承诺条款。

---

## 何时使用

- 用户要求撰写客户方案、提案材料、售前方案、解决方案说明
- 需要把客户需求、行业背景、竞品差异整合成完整正文
- 多 Skill 流程中，作为核心汇合节点输出提纲与正文

不单独用于：
- 只查市场信息或行业趋势 -> 使用市场洞察 Skill
- 只做竞品对比 -> 使用竞争分析 Skill
- 只做报价或预算拆分 -> 使用预算报价 Skill
- 只写一封发送邮件 -> 使用邮件回复 Skill

---

## 输入

**必需：**
- 客户名称或客户类型
- 需求描述、项目目标或待解决问题

**可选：**
- 客户背景材料、会议纪要、RFP、需求清单
- 市场洞察摘要
- 竞品分析摘要
- 预算范围、交付周期、已有资源约束
- 企业标准能力介绍、既有方案模板、案例素材

**信息不足时：**
- 缺少客户需求 -> 先输出需求理解框架，并标注待确认项
- 缺少交付边界 -> 使用保守表述，不承诺未确认内容

---

## 输出

一份 Markdown 格式的解决方案正文，默认包含以下部分：

1. **客户背景与目标**
2. **问题理解与需求拆解**
3. **方案设计**
4. **实施方式与阶段安排**
5. **预期价值**
6. **风险、边界与前提条件**
7. **下一步建议**

当用户需要时，可先输出：

- 一版方案提纲
- 一版摘要页
- 一版完整正文

---

## 核心流程

### 第一步：理解需求与边界

1. 提取客户名称、项目目标、核心问题、约束条件
2. 区分已确认信息、合理假设和待确认问题
3. 判断本次是内部讨论稿、对外正式版，还是仅需方案框架

### 第二步：整合上下文

1. 吸收市场洞察中的市场背景与机会判断
2. 吸收竞争分析中的差异化信息
3. 吸收预算报价中的成本或版本边界
4. 保留与客户当前诉求直接相关的部分，避免堆砌通用话术

### 第三步：搭建结构

1. 先形成提纲，明确章节顺序
2. 把“客户问题 -> 我们的回应 -> 实施方式 -> 风险边界”串成主线
3. 对未确认内容使用保守措辞或待确认占位

### 第四步：生成正文

1. 按结构展开正文
2. 优先写清问题理解、方案逻辑和实施路径
3. 若缺乏事实支撑，不空泛夸大，不虚构案例

### 第五步：输出与交接

1. 输出摘要、提纲或完整正文
2. 单独列出待确认问题，供后续补充
3. 若需要报价、发客户或做合规检查，则把正文交给后续 Skill

---

## 质量约束

- 不编造客户需求、成功案例或承诺条款
- 不把市场洞察、竞品对比原样堆砌成正文
- 不使用空泛套话替代问题理解
- 对未确认内容使用保守表述，如“建议”“拟”“可考虑”“待确认”
- 对外正式版要避免“保证”“一定”“无风险”等过满措辞

---

## 异常处理

| 情况 | 处理方式 |
|------|----------|
| 用户只给一句“写个方案” | 先输出提纲框架，并追问客户对象、目标和用途 |
| 需求频繁变化 | 保留最新版本为正文主线，旧信息转入待确认项 |
| 市场/竞品信息不足 | 只保留最小必要背景，正文聚焦需求理解与方案设计 |
| 用户只要内部讨论稿 | 降低修辞要求，重点保留问题、思路和风险边界 |
| 用户要求直接给终稿 | 仍应先检查信息完整性，不足则标注待确认而非硬写 |
