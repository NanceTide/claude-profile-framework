# 差旅报销 Skill

## Skill 目标

帮助用户完成企业内部差旅报销申请。识别报销意图，抽取报销所需信息，整理费用明细，生成规范化报销说明、系统表单字段或提交草稿，并在任何可能触达审批流或财务系统的动作前要求用户确认。

本 Skill 不替用户编造金额、票据、出差时间、出差事由、项目归属或审批对象。

## 何时使用

当用户表达以下或类似意图时使用：

- 帮我报销一下
- 把这次出差费用整理一下
- 这些发票帮我弄成报销单
- 帮我写个报销说明
- 帮我整理成能提交系统的版本
- 上周去上海那趟帮我走报销

当用户只是咨询报销制度、询问可报范围、询问流程，但尚未要求生成或提交申请时，先解释或确认，不直接进入提交流程。

## 输入

### 用户显式输入

优先从当前对话提取：

- 报销事由
- 出差时间范围
- 出差地点
- 费用明细
- 每笔费用金额
- 费用类别：交通、住宿、餐饮、市内交通、招待等
- 发票或票据情况
- 项目编号或成本归属
- 审批人
- 抄送人
- 输出形式：正式说明、表单字段、邮件、即时消息
- 是否需要提交、发送或转交

### 可从 Memory 读取的信息

- 用户姓名
- 工号
- 所属部门
- 岗位名称
- 成本中心
- 常用项目编号
- 常用审批人
- 常用抄送人
- 常用收款账户
- 常用报销主体
- 常用出发城市
- 企业报销系统入口
- 企业报销格式偏好

### 不得擅自编造的信息

以下信息缺失时必须确认，不得编造：

- 本次报销对应的时间范围
- 本次报销事由
- 本次费用明细
- 本次金额
- 本次票据状态
- 本次审批人
- 是否提交

## 输出

根据用户需求输出以下一种或多种结果。

### 报销说明文本

适用于发送给领导、财务或行政，包含：

- 报销事由
- 时间范围
- 费用概览
- 项目或成本归属
- 票据说明
- 审批请求

### 报销表单字段

适用于 OA、财务系统、ERP 或企业内部报销平台：

```yaml
报销人:
部门:
工号:
成本中心:
项目编号:
报销事由:
时间范围:
出差地点:
费用明细:
费用合计:
票据情况:
审批人:
抄送人:
收款账户:
备注:
```

### 费用明细清单

适用于先整理后提交的场景，建议按以下结构输出：

| 日期 | 类别 | 金额 | 说明 | 票据状态 |
|------|------|------|------|---------|

### 提交前确认

任何会触达审批人、财务或系统的动作前，必须展示摘要并等待用户确认：

```text
以下是我整理的报销申请信息，请确认是否提交：
……
请回复「确认提交」或指出需要修改的内容。
```

## 核心流程

1. **识别意图。** 判断用户是咨询制度，还是要生成或提交报销申请。
2. **抽取信息。** 从对话中抽取报销事由、时间范围、费用类别、金额、票据状态、审批信息等字段。
3. **读取 Memory。** 读取姓名、部门、工号、成本中心、常用项目编号、常用审批人等稳定信息。
4. **检查完整性。** 提交前至少确认报销人、部门、报销事由、时间范围、费用明细、费用合计、票据情况、审批人。
5. **生成草稿。** 按用户需要生成报销说明、表单字段、费用清单或沟通话术。
6. **提交前确认。** 若用户要求发送、提交或转交，必须先展示最终内容并等待确认。
7. **更新 Memory。** 若用户提供或更正了长期复用信息，则更新这些稳定事实；一次性报销内容不应写入 Memory。

## 质量约束

- 不编造金额、票据、项目归属、审批人。
- 金额不清楚时必须追问，不得猜测。
- 草稿允许用「待补充」占位；提交前不得保留关键字段空缺。
- 若票据缺失，应明确标注「待补票」或「待核验」，不假定已齐全。
- 报销理由保持真实、简洁、业务化，不过度扩写。
- 直接提交、发送、转交前必须得到用户明确确认。

## 异常处理

| 情况 | 处理方式 |
|------|---------|
| 用户只说「帮我报销一下」 | 追问对应哪次出差、费用类别、金额、票据情况 |
| 缺少工号/部门/成本中心 | 先从 Memory 取；取不到再问 |
| 只有零散票据，没有完整说明 | 先整理清单，再提示待补事由与时间范围 |
| 用户金额说得模糊 | 标「待确认」，不自动汇总成确定值 |
| 用户要求直接提交 | 展示最终内容，等待「确认提交」 |
| 票据不全 | 明确列入待补材料清单，不视为已完成 |
