# 长期饮食与备餐规划 Skill

## Skill 目标

根据用户的饮食目标、生活节奏、做饭条件和口味偏好，生成结构化、可执行、便于长期坚持的饮食与备餐计划。

本 Skill 不做医疗诊断，不在信息不充分时假定用户适合特定饮食法；涉及疾病、严重过敏或明显身体异常时，应提醒用户谨慎确认。

---

## 何时使用

- 用户要求安排一周饮食
- 用户希望做备餐计划、买菜计划、控体重饮食规划
- 用户想把“吃什么、什么时候准备、买什么”整合成可执行方案
- 用户希望根据近期状态调整饮食节奏

不单独用于：
- 只问一道菜怎么做
- 只问某个食材热量
- 只查某种营养学知识点

---

## 输入

**必需：**
- 至少一个明确目标或约束，例如：减脂、增肌、控糖、吃得清淡、省时间、省钱、规律吃饭

**可选：**
- 可规划时间范围（如 3 天、1 周）
- 做饭频率
- 厨房条件
- 预算范围
- 口味偏好
- 忌口或过敏项
- 常备食材
- 每日作息
- 近期特殊情况（如出差、训练增加、肠胃不适、聚餐）

**信息不足时：**
- 若用户目标很模糊，只说“帮我安排吃饭”，先收敛目标
- 若关键信息缺失较多，可先给基础版方案，并标注待确认项

---

## 输出

根据用户需求输出以下一种或多种结果：

1. **饮食安排概览**
   - 本周饮食目标
   - 计划风格（轻量、省事、均衡、控热量等）

2. **每日餐食建议**
   - 早餐 / 午餐 / 晚餐 / 加餐
   - 每餐核心思路，而不是只列菜名

3. **备餐安排**
   - 哪些适合提前准备
   - 哪些当天做更合适
   - 哪些可一份多吃

4. **购物清单**
   - 按蛋白质、蔬菜、水果、主食、调味和补充食材分类

5. **调整建议**
   - 时间不够时怎么降级
   - 状态变差时怎么减负
   - 外食时如何替代

6. **待确认项**
   - 当前方案成立但仍待用户确认的信息

---

## 核心流程

### 第一步：明确饮食目标与边界

1. 判断用户当前更看重什么：体重控制、健康、方便、省钱、稳定执行
2. 明确有无硬约束：过敏、忌口、作息、厨房条件、预算
3. 区分长期偏好与近期特殊情况

### 第二步：判断执行强度

1. 如果用户近期状态特殊，优先调整计划强度
2. 如果用户时间少，优先安排省事、可提前备餐的组合
3. 如果用户目标明确但执行难度高，优先给可坚持版本，而不是理想化版本

### 第三步：生成饮食方案

1. 先确定每日餐次结构
2. 再安排每餐的思路和搭配
3. 确保一周内食材和准备动作有复用，减少浪费和复杂度

### 第四步：生成备餐与采购安排

1. 列哪些食材适合统一采购
2. 列哪些内容适合提前处理
3. 控制总复杂度，避免“计划看起来很好但做不出来”

### 第五步：输出与调整

1. 输出可直接执行的计划
2. 如果方案依赖未确认信息，单列待确认项
3. 同时给出“忙的时候怎么做简化版”

---

## 质量约束

- 不把饮食计划写成空泛健康建议
- 不堆砌营养术语，优先可执行性
- 不假装用户有充足时间和完整厨房条件
- 不把每一天都安排得很满
- 优先考虑长期能坚持，而不是短期看起来完美
- 若涉及明显医疗风险，只能提示谨慎，不可替代专业建议

---

## 异常处理

| 情况 | 处理方式 |
|------|----------|
| 用户只说“最近吃得乱，帮我安排一下” | 先判断目标是减负、减脂还是恢复规律，再给基础版 |
| 用户目标互相冲突 | 直接指出冲突，例如“想省钱、极省时、每天高蛋白且完全不重复”无法同时最优 |
| 用户时间极少 | 优先给低复杂度饮食框架和备餐建议 |
| 用户近期身体状态特殊 | 降低计划强度，避免刺激性安排 |
| 用户厨房条件有限 | 方案改为可快速完成或可外食替代 |
| 用户临时有聚餐/出差 | 不推翻整周计划，只局部调整 |
